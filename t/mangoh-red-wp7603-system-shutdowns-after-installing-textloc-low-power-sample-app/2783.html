I am running my mangOH red (wp7603) dev board using Leaf with the latest firmware WP76xx Firmware Release 13.1 and Legato 19.02.0. My system was working fine until I build and updated my board with the textLoc (Low Power) sample app. After installing this app, my system now shutdowns (all LEDs off) and I cannot restart unless I disconnect the power and console USB cables and reconnect them back again. The system then comes back up, but it shutdowns again about 30 seconds after I get the login prompt in the USB console output. I assume it is related to the low power mode trigger from the textLoc app that I installed. So, I restarted the system again and I was able to login and run “app remove textLoc” before the system shutdown again. The problem is that I no longer see the app installed, but the system keeps shutting down.

Eventually I had to recover my system using the WPxxxx Windows recovery procedure. Initially the system was stable, until I updated back my mangOH board to the latest firmware. After the firmware update, I begun to experience again the same system shutdown with all LEDs off.

As I troubleshot my system, I was eventually able to notice that if I used the “cm radio off” command before the system shutdowns, then the system will not shutdown and will remain up without any issues. If I “cm radio on” again then the system begins to experience the shutdown issue. So, at this point I am wondering if the textLoc app that I installed, somehow enabled a cloud service to trigger my system low power shutdown. Can somebody clarify if this is what is happening and if so, how can I disable this low power trigger? Or alternatively, is this low power trigger hardwired in some internal register?

I was able to disable entering the ultra low power mode by modifying the “enable” file content located in /sys/module/swimcu_pm/boot_source

Note that I had to do this using the USB console port. Also, as I indicated before, when the system booted I had to immediately login and execute the “cm radio off” command. I am still not sure how is this related, but it avoided triggering the ulpm shutdown process.

Then I executed the following commands:

cd /sys/module/swimcu_pm/boot_source
echo 0 > enable
reboot

After the reboot, the system started and did not trigger the ulpm shutdown. So the system is now stable.
